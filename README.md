# Task Manager

## О проекте

Task Manager — тестовое приложение для [NDA]

## Стек

### Core

- React 19
- TypeScript 5.9
- Vite 7
- Redux Toolkit + RTK Query
- React Router DOM 6.30

### UI & Forms

- MUI 6
- React Hook Form + Zod

### Testing

- Vitest
- Storybook
- Playwright

### Tools & CI/CD

- ESLint + Prettier
- Husky + lint-staged
- GitHub Actions
- json-server (Mock API)

## Требования

- Node.js >= 22.12.0 и доступна в PATH
- npm >= 10.9.0 и доступен в PATH
- Git установлен и доступен в PATH

## Запуск проекта

```bash
git clone <repository-url>
cd task-manager
npm install
npm run dev
```

## Архитектура и описание

SPA на React + TypeScript, собранная через Vite, со стейт-менеджментом на Redux Toolkit и серверными запросами через RTK Query.
UI построен на MUI (Material UI), формы — React Hook Form + Zod.
Для разработки используется моковый REST API на json-server.
Компоненты и UI-сценарии документируются и тестируются через Storybook, а юнит/компонентные тесты запускаются через Vitest (включая Storybook tests в браузере через Playwright).

- Данные кешируются и инвалидируются в RTK Query
- Апишка разделена по логическим частям(базовая настройка, задача и тег)
- Данные валидируются как на входе от сервера, так и при пользовательском вводе
- По структуре папок: все построено от страниц, компоненты\типы\хелперы и т.д которые используются только на странице находятся внутри этой страницы, если переиспользуется - в shared. Отдельно созданы entities, там хранятся основные сущности и все данные для них. Это же касается и app
- Для github настроен CI который запускает линтеры, тайпчекеры, тесты и сборку. После этого можно делать CD, если нужно
- Добавлены pre-commit хуки через husky которые не дают запушить неотформатированный код или код с ошибками в измененных файлах
- Все роуты оформлены через константы и build ф-ции чтобы исключить ошибки и magic path
- Все константы тоже вынесены, никаких magic number
- Компонент просмотра данных задачи унифицирован, модифицирует свои возможности в зависимости от места нахождения(пропсов)
- Форма создания и редактирования тоже унифицирована
- Сделан глобальный Loader, сам следит когда есть запросы
- Важные части обернуты в ErrorBoundary для отлавливания ошибок
- При ошибке в запросах предусмотрен retry в виде кнопки и уведомления
- Поиск через like, с debounce
- Коммиты пишутся в стиле Conventional Commits чтобы потом можно было автоматически генерировать changelog при релизах + упрощает версионирование
